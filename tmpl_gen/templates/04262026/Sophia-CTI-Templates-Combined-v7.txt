================================================================================
SECTION A. Design history and rationale
================================================================================

1. The problem we are training for
----------------------------------
AthenaBench (https://athenabench.org) is a five-task CTI benchmark with a
small validation slice (~500 rows per task). The five tasks and their reading:

  athena-rcm  Reverse Campaign Mapping     CVE description -> CWE id
  athena-vsp  Vuln Scoring Profile         CVE description -> CVSS v3.1 vector
  athena-ate  ATT&CK Technique Extraction  threat report -> T#### technique ids
  athena-taa  Threat Actor Attribution     report fragment -> intrusion-set
  athena-rms  Risk Mitigation Strategy     attack scenario -> N x M#### ids

The Llama-3.1-8B-Instruct base scores poorly on all five out-of-the-box
(athena-rms baseline F1 = 5.88%). Improving these scores via full-parameter
SFT on a curated synthetic CTI corpus is the v7 program goal.

2. Corpus build pipeline
------------------------
Templates in this file are processed by tmpl_gen (see tmpl_gen/README.md and
tmpl_gen/templates/README.md). The pipeline is:

  .txt template      -> tmpl_docx2json.py        -> templates.json
  templates.json     -> iftgen.py + Neo4j        -> per-template triples
  per-tmpl triples   -> to_alpaca.py             -> Alpaca-format JSON

The Neo4j graph contains the MITRE ATT&CK, CWE, CVE, CAPEC, KEV, EPSS, and
ENGAGE knowledge bases plus several derived nodes. Each template selects a
subgraph pattern and a generation cardinality (Count: directive); iftgen
walks the graph and emits one row per match (capped by Count: or by
--count_max from the launcher).

3. Template family taxonomy
---------------------------
Every shortname in the v7 corpus belongs to one of the families below. The
families are kept stable across v0..v7 so that all SFT runs share a vocabulary
of skills.

  AB.*       AthenaBench-aligned templates (one or more per benchmark task)
             AB.RCM, AB.VSP, AB.ATE, AB.TAA, AB.RMS, AB.MCQ
  M.*        ATT&CK core (technique, tactic, campaign, malware, mitigation)
  A.*        CAPEC attack patterns + CWE/CVE/ATT&CK bridges
  W.*        CWE weakness analysis (root cause, detection, mitigation)
  V.*        CVE interpretation + CVSS triage
  S.*        CISA KEV operational urgency + remediation
  P.*        FIRST EPSS exploit-likelihood prioritization
  E.*        MITRE ENGAGE active-defense planning
  X.*        Multi-source cross-correlation chains (ATT&CK<->CAPEC<->CWE<->CVE
             <->KEV<->EPSS<->ENGAGE) -- the heaviest learners of joint reasoning
  Q.*        Question-only single-graph "quiz" templates (general MITRE QA)
  CL.*       Cleartext / disambiguation aids for the multi-source families
  SU.*       Summary aids that walk a single source from one entrypoint
  YN.*       Yes/no discriminative templates (positive and negative variants)
  POC.*      Proof-of-Concept exploit context (atomic-red-team-style)
  EDB.*      Exploit-DB record interpretation
  SR.*       Sigma rule / signature reasoning
  V.CPE.*    CVE -> CPE / vendor-product correlation
  AB.RMS.3*  v7 variable-N RMS scenarios (N=1..8) -- the cardinality fix
  AB.RMS.4   v7 mitigation catalog flashcard (ID -> name)
  AB.RMS.5   v7 mitigation catalog flashcard (name -> ID)
  AB.RMS.6   v7 negative-discrimination RMS template

4. Lineage v0 -> v7 (what failed at each step and why we kept iterating)
------------------------------------------------------------------------
v0   baseline, no SFT. athena-rms F1 = 5.88%. Loss path: model has the right
     M-IDs in distribution but wrong cardinality and wrong format.

v1-v3 incremental broad-CTI SFT runs. Improved athena-rcm/vsp/ate/taa, did not
     consistently improve athena-rms; some runs regressed it slightly.

v4   first AthenaBench-aligned (AB.*) addendum. AB.RCM and AB.VSP gained 5-15
     pp; AB.RMS unchanged because the addendum only had AB.RMS.1 (single-id)
     and AB.RMS.2 (pair) templates.

v5   broad-coverage release used as the trunk for v6/v7. 170,500 rows over 174
     templates. AB.* coverage expanded to 25 templates, X.*/YN.*/Q.* expanded
     to cover detection/exploit families. v5 is the "everything except an
     RMS-specific fix" baseline this combined corpus rests on.

v6   first attempted athena-rms fix. Added an RMS-only addendum (RMS.3 N=3-5,
     RMS.4/5 catalog flashcards, RMS.6 negative). Result: F1 dropped from
     5.88% to 0.00%. Post-mortem identified three structural bugs:
       (1) Output truncation: full {coa.description} per mitigation pushed
           N=4-5 outputs to ~1500 chars. Combined with the 04022026.txt
           inputs (~300-1000 tokens) and the v6 launcher's --cutoff 2048,
           ~80%+ of training rows were right-truncated mid-explanation.
       (2) Wrong terminator: the AthenaBench RMS prompt mandates a final
           line beginning with "Answer:" containing only the M-IDs. v6
           outputs ended with "Therefore, the recommended ..." and never
           emitted an Answer: line; the trained model emitted zero
           compliant responses.
       (3) Cardinality gap: v6 trained N=3,4,5 only. Benchmark distribution
           peaks at N=1 (39%) and N=2 (24%) -- v6 trained on neither. The
           trained model collapsed to emitting exactly one M-ID in 98.4% of
           rows, all from a small high-frequency anchor set (M1037 picked
           28% of the time).
     v6 is the version this combined corpus replaces.

v7   addresses all three v6 bugs. Adds the AB.RMS.3a..3h slate (N=1..8, one
     template per N, weighted to match benchmark mass), reduces per-mitigation
     output to "{coa.mitre_id} ({coa.name})" (no inline description), every
     variable-N template ends with a literal "Answer: M####, M####, ..."
     terminator line, instruction text aligned verbatim with the benchmark
     prompt. v7 launcher bumps --cutoff 2048 -> 4096 to give headroom.
     This required parser fixes:
       - tmpl_gen/scripts/tmpl_docx2json.py: multi-line Answer: bodies
         (so a template's Answer: section can include the literal trailing
         "Answer: M####" line).
       - tmpl_gen/scripts/to_alpaca.py: lazy quantifiers in the
         Instruction/Question regex (greedy match would absorb the
         reasoning body up to the trailing "Answer:" marker and discard it).

5. Combined-corpus composition
------------------------------
Training rows by source contribution (180,533 total):

   79,816 (44.2%)  v5 broad CTI               A/E/M/P/S/V/W/X.{1..8}    [Section C]
   62,099 (34.4%)  v5 benchmark addendum      X.*, Q.*, CL.*, SU.*, YN. [Section D]
   17,685 ( 9.8%)  v5 detections/exploits     EDB.*, POC.*, SR.*, CPE   [Section E]
   10,900 ( 6.0%)  v5 AthenaBench-aligned     AB.{ATE,MCQ,RCM,RMS,TAA,VSP} [Section B]
   10,033 ( 5.6%)  v7 RMS addendum            AB.RMS.{3a..3h,4,5,6}     [Section F]

Per-section template counts: B=25, C=64, D=43, E=42, F=11  (174 + 11 = 185 templates)

The v5 "broad CTI" + "benchmark addendum" + "detections/exploits" sections
together provide the representational substrate (CTI vocabulary, multi-source
joins, explanation patterns). The v5 AthenaBench-aligned section provides the
five-task answer formats. The v7 RMS addendum is the narrowly targeted fix
for the one task v5 failed to recover (athena-rms).

6. How to read the rest of this file
------------------------------------
Sections B through F below are the verbatim source template files used to
generate the combined v7 corpus, in the order they were generated. Each
section is delimited by an "==== SECTION X ====" banner and prefixed with the
source-file path so that any individual template can still be traced back to
its origin. The intra-section template syntax follows tmpl_gen/templates/
README.md (Instruction:/Question:/Answer:/Count: with {var:Type.prop} graph
placeholders).

AthenaBench-Aligned CTI Templates (Addendum to Sophia-CTI-Templates-04022026)

Revision 04222026: adds native MCQ option shuffling (Shuffle: mcq), expands
TAA coverage (AB.TAA.3-6), adds forward-direction ATE variants (AB.ATE.4-5),
and adds CVSS metric-by-metric VSP variants (AB.VSP.3-4).

AthenaBench task alignment:

  Section 1 - AB.MCQ.* : 5-option multiple choice (A-E)         -> athena-mcq
  Section 2 - AB.RCM.* : CVE description -> CWE-####            -> athena-rcm
  Section 3 - AB.VSP.* : CVE description -> CVSS:3.1/... vector -> athena-vsp
  Section 4 - AB.ATE.* : attack scenario -> T####               -> athena-ate
  Section 5 - AB.RMS.* : attack scenario -> M10xx (list)        -> athena-rms
  Section 6 - AB.TAA.* : incident summary -> threat actor name  -> athena-taa

Design notes:

  - MCQ templates declare "Shuffle: mcq" so iftgen randomises option labels
    per row and rewrites the final "Therefore, X." letter, removing the
    position-A bias observed in earlier SFT runs.
  - TAA expansion drops the single mandatory malware slot so templates bind
    against graph paths that exist densely (group_uses_tech=4362,
    group_uses_tool=457, group_uses_malware=647,
    campaign_attributed_to=25).
  - ATE expansion adds forward-direction (description -> T-ID) variants
    framed as telemetry narratives and data-source snippets, matching the
    AthenaBench athena-ate task direction.
  - VSP expansion adds metric-by-metric justification text so the model
    learns to reason about AV/AC/PR/UI/S/C/I/A before emitting the vector.

ID convention:

  AB.<TASK>.<N>   AthenaBench task, sequential index

Answer: The record describes {cwe:cve.problemType>Weakness.name}, whose defining characteristic is {cwe.description}. Therefore, the root cause is {cwe.id}.

Benchmark-Aligned CTI Templates (Addendum to Sophia-CTI-Templates-04022026)

This revision (2026-04-20) addresses two classes of errors reported against the
original addendum:

  1. Variable redefinition: per tmpl_gen/src/tmpl_gen/tmpl_parser.py, a variable
     is defined exactly once per template (Instruction + Question + Answer)
     using {var:Type.prop} or {var:existing.rel>Type.prop}. All subsequent
     references in the same template drop the ":Type" part.

  2. Schema alignment: ATT&CK mitigations are course-of-action nodes (not
     Mitigation); attack-pattern is reached from course-of-action via the
     "mitigates" relationship, so from attack-pattern we traverse the inverse
     ("mitigates<course-of-action"). CAPEC->Weakness uses "exploits" (lower-
     case). CAPEC->attack-pattern uses "map_ap". CWE->CAPEC uses
     "related_attack_pattern".

This template set still addresses the three SecKnowledge / CyberPal.AI gaps
(Levi et al., 2024, arXiv:2408.09304):

  Gap 1: Multi-hop cross-framework path templates
  Gap 2: Hard-negative / discrimination templates
  Gap 3: Task-type diversity (MCQ, Yes/No, Classification, Summarization)

ID convention:
  X.<edge>.<n>    cross-framework path  (e.g. X.VW.1 = CVE -> Weakness)
  YN.<edge>.P.n   yes/no positive
  YN.<edge>.N.n   yes/no hard-negative  (uses {force} to pick unconnected node)
  Q.<edge>.<n>    multiple-choice (A/B/C/D)
  CL.<src>.<n>    classification (closed-list)
  SU.<src>.<n>    summarization (long -> short)

Edge abbreviations:
  V = CVE          W = Weakness (CWE)      A = CAPEC
  M = Technique    T = Tactic              G = intrusion-set (threat group)
  N = course-of-action (ATT&CK mitigation) P = EPSS           K = KEV

Note on {force}: as-written, distractor selection picks random non-match nodes.
For SecKnowledge-style adversarial distractors (semantically near the correct
answer rather than random), extend iftgen.py to pre-embed node descriptions and
filter candidate bindings by cosine similarity to the correct answer.

Design notes:

  - The V.CPE.* templates treat cpe_matches as opaque JSON text at bind time
    and teach the model to parse CPE 2.3 strings
    (cpe:2.3:<part>:<vendor>:<product>:<version>:...) in the Answer, so
    the model learns to interpret the field rather than just echo it.
  - SR.* templates never bind the raw "detection" JSON into the Question
    (the rule logic is too noisy) but do use it in the Answer where the
    model is asked to summarise the detection logic. This keeps prompts
    short while still teaching rule interpretation.
  - EDB.* and POC.* use neo4j_datetime fields (published) rendered as
    strings; date arithmetic is out of scope for these templates.
  - TAA-style hard-negatives in YN.N.* use {force} on the primary key
    (id / mitre_id) to guarantee the distractor is genuinely unrelated.
  - All MCQ templates declare "Shuffle: mcq" so iftgen randomises option
    labels per row and rewrites the final "Therefore, X." letter.

AthenaBench-Aligned CTI Templates -- RMS Addendum v7 (companion to ift_data_2026_04_24_v5)

Revision 04262026-RMS-addendum-v7: pure RMS-only addendum intended to be
trained jointly with ift_data_2026_04_24_v5. Replaces the v6 RMS-addendum
slate (tmpl_gen/templates/04252026/Sophia-CTI-Templates-AthenaBench-abaligned-v6.txt)
after a v6 post-mortem identified three structural bugs that drove the
trained model to a worse-than-baseline athena-rms result:

  (1) Output truncation. v6 inlines the full {coa.description} per
      mitigation in the Answer body. At N=4-5 mitigations the output
      reaches ~1500 chars / ~400-700 tokens. Combined with the input
      attack-pattern description (~300-1000 tokens) and the v6
      launcher's --cutoff 2048, ~80%+ of training rows are
      right-truncated mid-explanation. The model never sees the
      terminal sentence and never learns where to stop.

  (2) Wrong terminator format. The AthenaBench RMS prompt mandates a
      final line beginning with "Answer:" containing only the
      mitigation IDs:
          "The final line of your response must begin with 'Answer:'
           and contain only the mitigation IDs ..."
      v6 outputs end with "Therefore, the recommended mitigations are
      M####, M####." and never emit an "Answer:" line. The trained
      model inherits this format and emits zero compliant responses.

  (3) Cardinality coverage gap. v6 ships templates for N=3,4,5 only.
      The benchmark distribution is N=1..10 with the heaviest mass at
      N=1 (39%) and N=2 (24%) -- v6 does not train on either. The
      trained model collapses to emitting exactly one M-ID per row in
      98.4% of rows.

What v7 changes vs v6:

  - Variable-N coverage spans N=1..8 (one template per N), matching the
    benchmark prompt distribution. N=9, N=10 omitted (<2% of benchmark
    each, Neo4j-cardinality-limited).
  - Each per-mitigation clause is reduced to "{coa.mitre_id} ({coa.name})"
    with no inline {coa.description}. Estimated output length at N=8 is
    <600 chars / ~150 tokens. Combined with the largest observed input
    (~1000 tokens) the full row fits comfortably under cutoff_len=4096
    (the v7 launcher bumps from 2048 to 4096 for headroom).
  - Every variable-N template terminates with a literal "Answer: M####,
    M####, ..." line. This requires the lazy-quantifier fix in
    tmpl_gen/scripts/to_alpaca.py (greedy match would absorb the
    reasoning body up to the trailing "Answer:" marker and discard it).
  - Instruction text is brought verbatim into alignment with the
    benchmark prompt language ("Return exactly N mitigation IDs as a
    comma-separated list. Briefly justify your choice. The final line
    of your response must begin with 'Answer:' ...").
  - RMS.4/RMS.5 catalog flashcards retained from v6 unchanged
    (single-mitigation flashcards do not need the Answer-line
    terminator and are scored on substring presence of the M-ID).
  - RMS.6 discriminative negative retained but extended with an
    "Answer: M####" terminator for format consistency with the
    variable-N templates.

Combined-training intent (continuation of Option B from the v0 -> v6 plan):

  - Training command consumes --dataset ift_data_2026_04_24_v5,
    ift_data_2026_04_26_abaligned_v7, alpaca_en_demo

    RMS share rises to ~5-6% of corpus.
  - Format conflict (v5 no-Answer-line vs v7 with-Answer-line) is
