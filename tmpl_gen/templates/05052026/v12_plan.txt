## 0. Why v12 exists (v11 post-mortem in one table)

v11 14B shipped at weighted total 54.3, missing 3 of 4 §8 pass criteria:
weighted (54.3 vs 56.4 v9 bar), RMS (48.0 vs 65.8 bar), and the
no-axis-regressed-3pp-vs-v10 rule (VSP -15.8 pp vs v10).

Root cause: the v11 build pipeline did not produce the planned row
counts. The §3.1 eval-aligned families came in at 22-86% of plan, the
new CM.* family shipped 0 rows, AB.TAA.* overshot 2.45x (eating other
families' budget), and the §6.3 stratified shuffle was never invoked.

| family       | v11 plan | v11 actual | % plan | bench impact                                    |
|--------------|---------:|-----------:|-------:|-------------------------------------------------|
| AB.RMS.*     |   13,000 |     11,181 |    86% | OK -- RMS regression traces to recipe, not data |
| AB.ATE       |   10,000 |      4,803 |    48% | ATE -10.2 vs v7; actually shrank vs v10         |
| AB.VSP.*     |   14,000 |      5,763 |    41% | VSP -15.8 vs v10; <1/2 of v10 row count         |
| AB.RCM       |   12,000 |      2,864 |    24% | RCM held by capacity, not corpus                |
| AB.MCQ       |    8,000 |      1,763 |    22% | feeds CKT alignment                             |
| AB.MS        |  paired  |      4,192 |     -  | MS templates produced rows; under-integrated    |
| AB.TAA.*     |    3,500 |      8,560 |   245% | OVERSHOT; ate ATE/VSP/RCM budget                |
| TAA.CANON.*  |    2,000 |        728 |    36% | TAA-accurate ceiling intact (16% vs 22% bar)    |
| SOC.*        |   12,000 |      4,834 |    40% | still produced +12.8 pp; latent headroom        |
| CM.*         |    6,000 |          0 |     0% | family entirely missing from build              |

What v11 did prove (preserve in v12):
- CKT trim worked: X.* / YN.* cuts -> CKT 57.3 -> 63.4 (+6.1 vs v7 14B).
- SOC.* reasoning generalises: 4.8K rows -> CyberSOCEval +12.8 pp.
- New external benches transfer: even partial SOC seed -> CyberMetric
  87.3 avg (modest +0.8 vs v10).

What v11 failed to prove (no signal due to corpus build):
- TAA.CANON family value -- 728 rows is below the memorisation floor.
- CM.* family value -- not authored at all.
- Stratified-shuffle effect on RMS dilution -- never deployed.
- Single-pass viability -- conflated with corpus underbuild.

## 2. Base-model decision

Production v12: Qwen2.5-14B-Instruct first; 32B held until 14B clears
v12 §8 pass criteria.

v11 explicitly proved "do not launch 32B blindly": v11 14B failed
pass criteria; running v11 32B as-is would have burned 12-14 h of
4xH100 to confirm a known-suboptimal recipe. v12 enforces the same
gate.

If v12 14B clears all §8 pass criteria, v12 32B launches with the same
two-phase recipe and the same v12 corpus.

Skip v12 on Llama-3.1-8B and Foundation-Sec-8B (held from v11).

Experimental fork: Qwen3-32B (no-think) gated on the harness reasoning-
parser fix landing first (same prerequisite as v11 plan §2).

## 4. Build pipeline

Mirrors v11 with renames + 3 new safety gates and 1 new shuffle phase.
New build dir _v12_build/.

```bash
# Phase 1 -- template -> triples -> Alpaca rows
bash tmpl_gen/data_generation/make_dataset.sh \
    tmpl_gen/templates/05052026/Sophia-CTI-Templates-v12.txt \
    _v12_build/triples \
    SFT/data/ift_data_2026_05_05_v12.raw.json \
    10 1500

# Phases 2-5 -- balance + dedup + ROW-COUNT GATE + STRATIFIED SHUFFLE
bash _v12_build/watcher.sh
```

### 4.1 NEW: row-count gate (Phase 4)

After dedup completes, the watcher invokes a new helper:

```bash
python tmpl_gen/scripts/check_corpus_row_counts.py \
    --input  SFT/data/ift_data_2026_05_05_v12.json \
    --plan   tmpl_gen/templates/05052026/v12_row_count_gate.json \
    --report _v12_build/row_count_gate_report.json
```

`v12_row_count_gate.json` lists each eval-aligned family with the
REJECT_IF_BELOW threshold from §3.1. If any family fails the gate, the
build halts before the train/val split, writes a per-family failure
report, and does NOT advance to Phase 5. This is the corpus-side
equivalent of v11's wandb eval-loss visibility gap: it makes the v11
silent failure loud at build time.

### 4.2 NEW: stratified shuffle (Phase 5)

After the row-count gate passes, the watcher invokes the existing
(but never-wired-in-v11) shuffler:

```bash
python tmpl_gen/scripts/stratified_shuffle.py \
    --input  SFT/data/ift_data_2026_05_05_v12.json \
    --output SFT/data/ift_data_2026_05_05_v12.shuffled.json \
    --seed 42 --validate
```

Then the val-slice builder reads the shuffled file (so the train/val
split inherits the stratified order), and the train json passed to
LLaMA-Factory has guaranteed family interleaving in every N-row
window.

### 4.3 NEW: AB.TAA total-cap (in actor-balance pass)

`taa_actor_balance.py` gains a `--max-rows-per-family-total` flag.
For v12, set 3500 for AB.TAA.* (matches §3.1 plan ceiling). Without
this, the v11 overshoot (8,560 rows from ACTOR_CAP=40 alone) repeats.
The per-actor cap remains 40; the total cap is the second-pass safety
net.

Build-side gencfg defaults for v12 (unchanged from v11):

| gencfg key            | v11 default | v12 default | rationale                                |
|-----------------------|-------------|-------------|------------------------------------------|
| per_primary_grouping  | true        | true        | sample-with-replacement anchor diversity |
| allow_nullprops       | true        | true        | tolerate null Neo4j property values      |

`_v12_build/watcher.sh` tunables:

| knob                  | v11 | v12  | rationale                                                    |
|-----------------------|----:|-----:|--------------------------------------------------------------|
| ACTOR_CAP             |  40 |   40 | hold; per-actor cap                                          |
| ACTOR_FLOOR           | 100 |  100 | hold; minimum distinct actors                                |
| TAA_TOTAL_CAP         |   - | 3500 | NEW: per-family total cap, enforces §3.1 budget              |
| DEDUP_HIT_THRESHOLD   |   1 |    1 | hold                                                         |
| DEDUP_DROP_THRESHOLD  |  50 |   50 | hold                                                         |
| ROW_COUNT_GATE        |   - |   on | NEW: per-§4.1                                                |
| STRATIFIED_SHUFFLE    |   - |   on | NEW: per-§4.2                                                |

Final dataset: SFT/data/ift_data_2026_05_05_v12.shuffled.json
(target 220-235K rows post all phases).

## 6. Training launchers (three-phase, reverts to v9-shaped architecture)

v11 single-pass at cutoff 8192 lost the RMS catalog supervision that v9's
two-phase Phase B preserved (v9 RMS=65.8, v11 RMS=48.0 on a similar
RMS slice). v12 reverts to a phased recipe with a third phase added for
TAA.CANON memorisation -- the only family that v11 proved is recipe-
sensitive and not corpus-sensitive.

### 6.1 14B production -- SFT/autotrain/run_sft_qwen25_14b_v12.sh

Three-phase, branched from run_abaligned_sft_qwen25_14b_v9.sh with the
v12 corpus splits and the new Phase C added. Only Phase C's merged
checkpoint is pushed to ${HF_USERNAME}/athena-cti-sft-qwen25-14b-v12.

Phase A -- broad knowledge anchor.
- Datasets   : ift_data_2026_05_05_v12_broad,tulu_3_sft_mixture,alpaca_en_demo
  (broad split = the v12 corpus minus the eval-aligned families consumed
   in Phase B and minus TAA.CANON.* consumed in Phase C; built by the
   new tmpl_gen/scripts/split_corpus_for_phases.py utility)
- 1 epoch, lr 1e-5, cutoff 8192, packing ON
- Effective batch 16
- save/eval every 500 steps

Phase B -- RMS + ATE + VSP + RCM catalog drills (v9-shape recovery).
- Datasets   : ift_data_2026_05_05_v12_rms_ate_vsp_rcm
  (~42K rows: AB.RMS.* + JS.RMS.* + AB.ATE + JS.ATE + AB.VSP.* + V.CPE
   + AB.RCM + JS.RCM + X.VW + YN.VW)
- 1 epoch, lr 5e-6, cutoff 16384, packing OFF
- Effective batch 8
- save/eval every 400 steps
- --model points at Phase A's output dir

Phase C -- TAA canonical-name memorisation (NEW).
- Datasets   : ift_data_2026_05_05_v12_taa_canon
  (~10K rows: TAA.CANON.{1,2,3} = alias->canonical + canonical->TTPs +

- 1 epoch, lr 3e-6, cutoff 8192, packing ON

- --model points at Phase B's output dir
- Push merged checkpoint to ${HF_USERNAME}/athena-cti-sft-qwen25-14b-v12

### 6.3 Corpus splitter (new utility)

tmpl_gen/scripts/split_corpus_for_phases.py -- reads the stratified-
shuffled v12 corpus (output of §4.2) and the val slice (§7), then emits
three disjoint training shards:

```bash
python tmpl_gen/scripts/split_corpus_for_phases.py \
    --input  SFT/data/ift_data_2026_05_05_v12.shuffled.json \
    --val    SFT/data/ift_data_2026_05_05_v12_val.json \
    --out-broad      SFT/data/ift_data_2026_05_05_v12_broad.json \
    --out-rms-ate-vsp-rcm SFT/data/ift_data_2026_05_05_v12_rms_ate_vsp_rcm.json \
    --out-taa-canon  SFT/data/ift_data_2026_05_05_v12_taa_canon.json \
    --report _v12_build/phase_split_report.json
```

Split rules (derived from row-level shortname, same axis rules as
build_val_slice.py):
- Phase B shard: AB.RMS.* + JS.RMS.* + AB.ATE + JS.ATE + AB.VSP.* + V.CPE
  + AB.RCM + JS.RCM + X.VW + YN.VW
- Phase C shard: TAA.CANON.*
- Phase A shard: everything else (broad knowledge + SOC + CM + remaining
  AthenaBench families that are NOT in Phase B or C)

Each shard is registered as its own dataset_info.json entry (one shard
per phase keeps the LLaMA-Factory --dataset arg simple and avoids
multi-shard sampling weights).

### 6.4 Stratified shuffle and per-phase shuffles

The per-phase splitter preserves the §4.2 stratified order within each
shard. LLaMA-Factory's per-epoch shuffle then operates over a shard
that is already pre-stratified. Phase B and C shards are family-narrow
by construction (RMS+ATE+VSP and TAA.CANON respectively); the
stratification on those phases ensures the small number of distinct
shortnames within each shard is interleaved evenly.

## 7. Held-out validation slice

SFT/data/ift_data_2026_05_05_v12_val.json -- ~50 rows per AthenaBench
axis (RMS, ATE, VSP, RCM, MCQ, TAA-attribute, TAA-IE, TAA-NEG, MS,
TAA-CANON, SOC, CM). Built by tmpl_gen/scripts/build_val_slice.py
(extended from the v11 version: add SOC and CM axis rules, add
TAA-CANON axis rule already present), reading from the row-count-gated
clean corpus BEFORE the stratified shuffle so the per-axis sampling is
deterministic given --seed.

Wired into all three phases via --eval_dataset so the run logs per-axis
eval loss every {500, 400, 200} steps. The Phase B and Phase C eval
runs only score the axes those phases train on (RMS/ATE/VSP for B,
TAA-CANON for C); Phase A scores all axes.

Per-axis eval loss visibility was the v11 plan's §7 deliverable that
landed but produced silent regressions because the build undershot.
With v12's §4.1 row-count gate plus the per-axis eval loss, both the
build and the training surface failures loudly.

## 8. Evaluation plan

After all three phases complete:
1. Push merged Phase C checkpoint to
   ${HF_USERNAME}/athena-cti-sft-qwen25-{14b,32b}-v12.
2. Register vLLM aliases athena-cti-sft-qwen25-{14b,32b}-v12-vllm in
   SFT/test/pipelines/models.py (mirrors v11 entries; max-len 32768
   for both 14B and 32B per the Qwen2.5 native ctx ceiling).
3. Run full AthenaBench + CyberSOCEval + CyberMetric (sizes 80, 2000,
   10000) suites via SFT/test/utils/serve_and_bench.sh.
4. Run the new athena-cti-taa-canonical bench (§10) to score the
   TAA.CANON family in isolation. v11's TAA-accurate +3 pp gain was
   confounded with the AB.TAA overshoot; v12 separates the two signals.
5. Compare v12 vs v11 vs v9 vs v7 on:
   - per-axis accuracy (RMS, ATE, CKT, RCM, VSP, MCQ, MS,
     TAA-attribute, TAA-IE, TAA-NEG, TAA-CANON)
   - weighted total (parity with the AthenaBench summary script)
   - CyberSOCEval combined accuracy
   - CyberMetric 80 / 2000 / 10000 averages
6. Publish per-phase intermediate checkpoints (Phase A only, Phase A+B)
   ONLY if Phase C ablation is required; default is publish merged Phase C
   to keep the HF repo single-checkpoint.

Pass criteria for v12 14B production (all four must hold):
- weighted total >= max(v9=56.4, v11=54.3) + 1.5 pp = 57.9
- RMS >= v9 RMS (65.8); v11's 48.0 must be recovered by Phase B
- ATE no regression > 3 pp vs v7 (52.6); v11's 42.4 must be recovered
- VSP no regression > 3 pp vs v10 (86.7 from §0 baseline); v11's 70.9
  must be recovered
- TAA-attribute >= 22% (carried from v11 §8); TAA-CANON slice score
  reported separately as a new axis
- No axis regressed > 3 pp vs v11 on the families v11 won (CKT, SOC,
  CM, RCM)

Fallback if Phase C over-fits TAA at the cost of other families: drop
Phase C entirely and ship Phase B output; treat TAA.CANON as a v12.1
follow-up. The Phase A+B split is the v9-shape recovery and is the
minimum-viable v12.

Fallback if Phase B over-corrects RMS at the cost of CKT (the v8.1
failure mode): re-introduce a small SOC + CM slice into Phase B (~2K
rows total) to dilute the RMS catalog supervision. Do NOT increase
Phase B lr; that is the v8.1 failure path.

## 9. Hyperparameter notes (carried from v9/v11 unless flagged)

- LR schedule: cosine, 5% warmup per phase, bf16. Per-phase LRs are
  1e-5 / 5e-6 / 3e-6 (A/B/C), matching v9 Phase A/B and adding a third
  decay step for the memorisation-only Phase C.
- Sequence packing: ON in A and C, OFF in B. v9 confirmed Phase B
  packing OFF was load-bearing for RMS catalog learning (each catalog
  example needs its own loss target, not shared with other rows).
- DeepSpeed ZeRO-3 with auto offload on <4 GPUs (14B), forced offload
  on (32B). Same DS config files as v9: ds_z3_config.json (no offload),
  ds_z3_offload_config.json (offload). No change.
- save_total_limit 2 per phase, save_only_model True (matches v9).
  Phase A/B intermediate dirs are NOT pushed to HF; only Phase C
