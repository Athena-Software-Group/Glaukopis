================================================================================
SECTION A. v7 -> v8 motivation
================================================================================

The v7 32B SFT model holds a +20 pp lead over the base on AthenaBench but
regresses ~12 pp on CyberSOCEval and ~6 pp on CyberMetric. Sample inspection
of v7 responses on the CyberSOCEval malware-TI task identified that the
trained model:

  - Emits free-form prose where the eval harness expects a top-level JSON
    object with specific keys ({"correct_answers": [...]} or
    {"behaviors": [...]}).
  - Repeats the AthenaBench-aligned narrative pattern ("MITRE ATT&CK
    classifies ... Therefore, X.") even when the prompt asks for a JSON
    blob, producing 0% strict-format matches on JSON-graded tasks.
  - Loses the CyberMetric MCQ shape ("Answer: X") in roughly 6% of rows,
    substituting v7-style "Therefore, X." closers.

v8 addresses format collapse with a parallel JSON-output corpus that mirrors
the v7 narrative templates one-for-one in graph coverage but emits a
top-level JSON object instead of free-form prose. Trained jointly with the
v7 corpus (see Phase A in the v8 launcher) the model retains the v7 prose
behaviour for AthenaBench and gains the JSON-emitting behaviour for
CyberSOCEval and any production tool-call pipeline.

================================================================================
SECTION B. Sentinel encoding for literal JSON characters
================================================================================

The tmpl_gen template grammar (tmpl_gen/src/tmpl_gen/tmpl_parser.py
make_templ_grammar1) reserves the characters `{`, `}`, `[`, `]` for qfield
and listsection delimiters and provides no escape mechanism in TEXTSECTION
text (rule TEXTSECTION = `/(?:[^{<[\]]|<(?!\*))+/`). Templates that need to
emit literal JSON braces or brackets in their Answer body therefore use
the following sentinel substitutions, which are post-processed to literal
characters by tmpl_gen/scripts/to_alpaca.py (--unescape-json-sentinels,
default on as of the v8 release):

    <OBR>   ->  {     open  brace
    <CBR>   ->  }     close brace
    <OBK>   ->  [     open  bracket
    <CBK>   ->  ]     close bracket

These four sentinels survive the parser because each begins with `<X` where
X is not `*`, satisfying the second alternative of the TEXTSECTION rule
(`<(?!\*)`), and the remaining characters are all outside the excluded set
`[^{<[\]]`. They are short, readable, unlikely to appear in CTI source
data, and reversible by a single str.replace call per sentinel.

The sentinels are NEVER used inside qfields, scopes, or constraints --
those are parsed normally and emit verbatim graph values into the output
text. Only TEXTSECTION-level literal JSON characters are sentineled.

SECTION C.1 - Templates dropped pre-generation (JS.MCQ.4, JS.MCQ.6)
================================================================================

Two templates from the original 20-template slate were dropped after the
2026-04-29 smoke test against the local athena-cti-db Neo4j instance.

JS.MCQ.4 (CWE -> CAPEC):  Question of the form "Which CAPEC primarily
exploits {cwe}? A) {related CAPEC} B-E) 4 unrelated CAPECs". The
generator query has a 4-way unrelated-CAPEC cartesian (~559^4 = 9.7e10
candidate rows) AND an outer ORDER BY rand() over RETURN DISTINCT,
which forces Neo4j to materialise the full DISTINCT result set before
sorting. With cwe.description (~1-2 KiB per row) projected through the
expansion, peak transaction memory exceeds the local 16 GiB
dbms.memory.transaction.total.max ceiling and the query OOMs. The
bounded LIMIT-before-RETURN fallback in tmpl_parser.py also returns 0
rows because the planner pre-evaluates the WHERE-filtered cartesian
before applying LIMIT.

JS.MCQ.6 (CAPEC -> ATT&CK):  Same shape as JS.MCQ.4, with a 4-way
unrelated-attack-pattern cartesian (~600^4 = 1.3e11) and primary
sampled from the ~270 CAPECs that have a :map_ap edge. Same OOM
mechanism, same 0-row fallback.

Workarounds considered and rejected:
  - per_primary_grouping=true gencfg (CALL subquery per primary):
    succeeds in isolation but increases per-template wall time from
    ~10s to ~50 min for a 500-row template, blowing the end-to-end
    generation budget for the whole slate.
  - Stripping description columns from RETURN: did not avoid the OOM
    because the cartesian itself (regardless of column width) is the
    memory driver under ORDER BY rand().
  - Re-architecting tmpl_gen to use a two-stage pre-sample of negative
    distractors (collect() into a list, then unwind 4 random elements
    per primary): out of scope for the v8 release; tracked as a future
    improvement to tmpl_parser.py.

Coverage impact:  CWE/CAPEC and CAPEC/ATT&CK MCQ relationships remain
covered by the AthenaBench-aligned narrative templates AB.MCQ.{2,4,6}
(present in the Phase A combined corpus), so v8 retains exposure to
those mappings -- only the JSON-output variants of those two specific
mappings are absent.
