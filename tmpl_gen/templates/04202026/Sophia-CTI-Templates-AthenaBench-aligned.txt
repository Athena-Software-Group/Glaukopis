AthenaBench-Aligned CTI Templates (Addendum to Sophia-CTI-Templates-04022026)

This addendum targets the six AthenaBench CTI evaluation tasks directly. Each
section mirrors the input/output contract of one benchmark task so that the
resulting instruction triples shape the SFT model to answer in the exact
format the benchmark expects.

AthenaBench task alignment:

  Section 1 - AB.MCQ.* : 5-option multiple choice (A-E)         -> athena-mcq
  Section 2 - AB.RCM.* : CVE description -> CWE-####            -> athena-rcm
  Section 3 - AB.VSP.* : CVE description -> CVSS:3.1/... vector -> athena-vsp
  Section 4 - AB.ATE.* : attack scenario -> T####               -> athena-ate
  Section 5 - AB.RMS.* : attack scenario -> M10xx (list)        -> athena-rms
  Section 6 - AB.TAA.* : incident summary -> threat actor name  -> athena-taa

Design notes:

  - Instruction text mirrors the AthenaBench prompt voice (persona, task
    framing, reasoning-before-answer) so inference-time prompts are in
    distribution.
  - Answers are reason-first: the analyst narrative comes first, the final
    label (letter / identifier / vector / name) comes last after "Therefore,".
  - Graph-stored identifiers are already prefixed (CVE.id = "CVE-...",
    Weakness.id = "CWE-..."). These templates reference them as {cve.id},
    {cwe.id}, {cve.problemType.id} without prepending a literal prefix, so
    output matches AthenaBench answer formatting exactly.
  - ATE/RMS/TAA "scenarios" use node .description fields verbatim. Replacing
    these with LLM-rewritten incident-report prose is deferred future work.
  - VSP uses the CVE CVSS v3.1 properties (cvss3_1_vectorString,
    cvss3_1_baseScore, cvss3_1_attackVector) populated by the CVE ingest.

ID convention:

  AB.<TASK>.<N>   AthenaBench task, sequential index

Section 2 - AB.RCM.* Root-Cause Mapping (athena-rcm)

AB.RCM.1 Instruction: You are a CTI analyst at an enterprise Security Operations Center performing root-cause mapping. Analyse the following CVE description and identify the most appropriate Common Weakness Enumeration (CWE) that represents the root cause of the vulnerability. Consult the MITRE CVE and CWE knowledge bases, reason step by step about the weakness pattern present in the description, and then state the CWE identifier.
Question: {cve:CVE.descriptions}
Answer: The description indicates the class of weakness captured by {cwe:cve.problemType>Weakness.name}: {cwe.description}. Therefore, the root cause is {cwe.id}.

AB.RCM.2 Instruction: You are a vulnerability analyst at an enterprise Security Operations Center performing root-cause mapping for a newly disclosed CVE. Analyse the vulnerability record below and identify the most appropriate CWE. Consult the MITRE CVE and CWE knowledge bases, reason about the underlying weakness before committing to a label, and then state the CWE identifier.
Question: Vulnerability record for {cve:CVE.id}: {cve.descriptions}
Answer: The record describes {cwe:cve.problemType>Weakness.name}, whose defining characteristic is {cwe.description}. Therefore, the root cause is {cwe.id}.

Section 4 - AB.ATE.* Attack Technique Enumeration (athena-ate)

AB.ATE.1 Instruction: You are a threat hunter at an enterprise Security Operations Center performing attack technique enumeration. Analyse the following attack scenario and identify the single MITRE ATT&CK technique (ID format: T####) that best matches the behaviour described. Consult MITRE ATT&CK, reason about observable indicators in the scenario, and then state the technique identifier.
Question: {ap:attack-pattern.description}
Answer: The described behaviour matches {ap.name}, which MITRE ATT&CK documents on platforms {ap.x_mitre_platforms} with detection guidance: {ap.x_mitre_detection}. Therefore, the technique is {ap.mitre_id}.

AB.ATE.2 Instruction: You are a detection engineer at an enterprise Security Operations Center performing attack technique enumeration from telemetry narratives. Analyse the following scenario and identify the single MITRE ATT&CK technique (ID format: T####) that best matches. Consult MITRE ATT&CK, reason about the adversary action implied by the narrative, and then state the technique identifier.
Question: {ap:attack-pattern.description}
Answer: The action sequence in the narrative is the defining behaviour of {ap.name}, collected from data sources {ap.x_mitre_data_sources}. Therefore, the technique is {ap.mitre_id}.

AB.ATE.3 Instruction: You are an incident responder at an enterprise Security Operations Center correlating attacker activity to MITRE ATT&CK. Analyse the following incident narrative, which reflects adversary tradecraft attributed to a documented threat group. Consult MITRE ATT&CK, reason about which technique underpins the observed behaviour, and then state the technique identifier (ID format: T####).
Question: During an engagement attributed to {grp:intrusion-set.name} ({grp.mitre_id}), analysts observed the following adversary behaviour: {ap:grp.uses>attack-pattern.description}
Answer: The observed behaviour is the documented pattern for {ap.name}, which MITRE ATT&CK records {grp.name} as using. Therefore, the technique is {ap.mitre_id}.

Section 6 - AB.TAA.* Threat Actor Attribution (athena-taa)

AB.TAA.1 Instruction: You are a CTI attribution analyst at an enterprise Security Operations Center performing threat actor attribution from observed tradecraft. Analyse the following anonymised incident summary, which reports the MITRE ATT&CK techniques and malware family observed during an intrusion, and identify the responsible named intrusion set. Consult the MITRE ATT&CK groups knowledge base, reason about which documented intrusion set the combination of techniques and tooling matches, and then state the adversary name.
Question: <*{grp:intrusion-set.name}*>An adversary has been observed employing MITRE ATT&CK techniques {ap1:grp.uses>attack-pattern.name} ({ap1.mitre_id}), {ap2:grp.uses>attack-pattern.name} ({ap2.mitre_id}), and {ap3:grp.uses>attack-pattern.name} ({ap3.mitre_id}), alongside the malware family {mw:grp.uses>malware.name}.
Answer: The joint use of {ap1.name}, {ap2.name}, and {ap3.name} together with {mw.name} matches the documented tradecraft of {grp.name} ({grp.mitre_id}). Therefore, the adversary is {grp.name}.
Sample: grp

Answer: Use of {ap1.name} and {ap2.name} together with {mw.name} matches the documented tradecraft of {grp.name} ({grp.mitre_id}). Therefore, the adversary is {grp.name}.
