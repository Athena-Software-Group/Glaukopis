AthenaBench-Aligned CTI Templates -- Trimmed (derived from Sophia-CTI-Templates-AthenaBench-aligned.txt)

Revision 04222026-trimmed: drops TAA coverage (athena-taa retired from
benchmark), thins the remaining sections by removing paraphrastic template
variants, and caps MCQ fanout to keep the balanced MCQ signal in the
training set without letting it dominate. Targets ~1k MCQ rows (down
from ~20k in the parent) so the model learns the answer format without
collapsing onto a position-A prior.

AthenaBench task alignment (trimmed slate):

  Section 1 - AB.MCQ.* : 5-option multiple choice (A-E)         -> athena-mcq
  Section 2 - AB.RCM.* : CVE description -> CWE-####            -> athena-rcm
  Section 3 - AB.VSP.* : CVE description -> CVSS:3.1/... vector -> athena-vsp
  Section 4 - AB.ATE.* : attack scenario -> T####               -> athena-ate
  Section 5 - AB.RMS.* : attack scenario -> M10xx (list)        -> athena-rms

What was removed vs Sophia-CTI-Templates-AthenaBench-aligned.txt:

  - AB.TAA.1-6 : entire section dropped (athena-taa retired from benchmark).
  - AB.VSP.2   : paraphrase of AB.VSP.1 (record-framed wording, same CVSS
                 derivation) -- dropped.
  - AB.VSP.4   : paraphrase of AB.VSP.3 (same per-metric reasoning, record
                 framing) -- dropped.
  - AB.ATE.2   : paraphrase of AB.ATE.1 (raw description -> T-ID) with
                 telemetry-narrative framing but identical binding --
                 dropped, coverage preserved by AB.ATE.5.
  - AB.ATE.4   : paraphrase of AB.ATE.1 with "Scenario:" framing but
                 identical binding -- dropped.

What was kept:

  - AB.MCQ.1-6 : all six retained for coverage breadth, each capped at
                 Count: 170 (~1020 total MCQ rows after shuffler balances
                 the correct-letter distribution uniformly across A-E).
  - AB.RCM.1-2 : 2 distinct CVE->CWE variants (description-only, full record).
  - AB.VSP.1,3 : headline score+vector + per-metric reasoning variants.
  - AB.ATE.1,3,5 : raw description, group-attributed narrative, telemetry-
                   framed. Each binds a meaningfully different graph path.
  - AB.RMS.1-2 : multi-mitigation list and single-primary-mitigation.

MCQ position-bias control:

  - Each MCQ template declares "Shuffle: mcq" so iftgen randomises option
    labels per row and rewrites the final "Therefore, X." letter. This
    must run against the fixed _shuffle_mcq_options in tmpl_parser.py
    (anchors the option block to the "Therefore, X." tail instead of a
    global count-all pass that silently bailed on ~50% of rows when CTI
    description text contained incidental "A) ... B) ..." bullet markers).
  - Each MCQ template declares "Count: 170" so the per-template Count
    directive caps fanout independent of the CLI --count_max ceiling.

Expected dataset effect:

  - Templates drop 25 -> 15 (a 40% reduction in template count).
  - Row count target: ~15k non-MCQ + ~1k MCQ = ~16k total.
  - SFT wall-clock on 4xH100 with batch=4 grad_accum=1: target ~3-4h
    (down from ~8.5h on the full 138k-row slate).

Template IDs retain their original numbering (AB.VSP.3, AB.ATE.5, etc.)
so diff-vs-parent traceability is preserved; there is no renumbering.

Section 2 - AB.RCM.* Root-Cause Mapping (athena-rcm)

AB.RCM.1 Instruction: You are a CTI analyst at an enterprise Security Operations Center performing root-cause mapping. Analyse the following CVE description and identify the most appropriate Common Weakness Enumeration (CWE) that represents the root cause of the vulnerability. Consult the MITRE CVE and CWE knowledge bases, reason step by step about the weakness pattern present in the description, and then state the CWE identifier.
Question: {cve:CVE.descriptions}
Answer: The description indicates the class of weakness captured by {cwe:cve.problemType>Weakness.name}: {cwe.description}. Therefore, the root cause is {cwe.id}.

AB.RCM.2 Instruction: You are a vulnerability analyst at an enterprise Security Operations Center performing root-cause mapping for a newly disclosed CVE. Analyse the vulnerability record below and identify the most appropriate CWE. Consult the MITRE CVE and CWE knowledge bases, reason about the underlying weakness before committing to a label, and then state the CWE identifier.
Question: Vulnerability record for {cve:CVE.id}: {cve.descriptions}
Answer: The record describes {cwe:cve.problemType>Weakness.name}, whose defining characteristic is {cwe.description}. Therefore, the root cause is {cwe.id}.

Consult the CVSS v3.1 specification, reason about each metric in turn, and then state the resulting CVSS:3.1 vector.
Question: {cve:CVE.descriptions}
Answer: The exploit preconditions and impact implied by the description correspond to a CVSS v3.1 base score of {cve.cvss3_1_baseScore}. Therefore, the CVSS v3.1 vector is {cve.cvss3_1_vectorString}. <*{force cve.cvss3_1_vectorString = cve.cvss3_1_vectorString}*>

AB.VSP.3 Instruction: You are a vulnerability risk analyst at an enterprise Security Operations Center producing a CVSS v3.1 Base vector with per-metric justification. Analyse the following CVE description, reason explicitly about each of the eight base metrics (Attack Vector, Attack Complexity, Privileges Required, User Interaction, Scope, Confidentiality, Integrity, Availability), and then state the resulting CVSS:3.1 vector.
Question: {cve:CVE.descriptions}
Answer: Reasoning through each base metric: Attack Vector of {cve.cvss3_1_attackVector} is indicated by how the attacker reaches the vulnerable component; Attack Complexity is implied by the conditions required to weaponise the flaw; Privileges Required and User Interaction follow from whether authentication or victim action is needed; Scope, Confidentiality, Integrity, and Availability follow from the impact described. The resulting CVSS v3.1 base score is {cve.cvss3_1_baseScore}. Therefore, the CVSS v3.1 vector is {cve.cvss3_1_vectorString}. <*{force cve.cvss3_1_vectorString = cve.cvss3_1_vectorString}*>

Section 4 - AB.ATE.* Attack Technique Enumeration (athena-ate)

Question: {ap:attack-pattern.description}
Answer: The described behaviour matches {ap.name}, which MITRE ATT&CK documents on platforms {ap.x_mitre_platforms} with detection guidance: {ap.x_mitre_detection}. Therefore, the technique is {ap.mitre_id}.

AB.ATE.3 Instruction: You are an incident responder at an enterprise Security Operations Center correlating attacker activity to MITRE ATT&CK. Consult MITRE ATT&CK, reason about which technique underpins the observed behaviour, and then state the technique identifier (ID format: T####).
Question: During an engagement attributed to {grp:intrusion-set.name} ({grp.mitre_id}), analysts observed the following adversary behaviour: {ap:grp.uses>attack-pattern.description}
Answer: The observed behaviour is the documented pattern for {ap.name}, which MITRE ATT&CK records {grp.name} as using. Therefore, the technique is {ap.mitre_id}.

Answer: The telemetry pattern matches {ap.name}, whose defining behaviour is captured by those data sources. Therefore, the technique is {ap.mitre_id}.
