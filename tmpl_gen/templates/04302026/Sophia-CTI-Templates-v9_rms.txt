================================================================================
Sophia CTI Templates -- v9 RMS slice (Qwen2.5-14B-Instruct, two-phase SFT Phase B)
================================================================================

This file is the self-contained template manifest for the v9 RMS catalog
drills. It is the AB.RMS.* + JS.RMS.* subset of the v8.1 manifest
(tmpl_gen/templates/04302026/Sophia-CTI-Templates-v8_1.txt) extracted as
its own first-class template file so the v9 build pipeline has a single,
auditable source of truth that does not depend on any v8.1 output
artefact.

The 21 template bodies below are byte-for-byte copies of the
corresponding bodies in v8_1.txt (Section F lines 1598-1761 for AB.RMS.*,
and Section D.4 lines 2014-2138 for JS.RMS.*). Keeping them inlined
rather than included means a grep for any AB.RMS or JS.RMS shortname in
the v9 corpus resolves to a single self-contained definition here, and
the v9 build can run even if v8.1's manifest is later edited or moved.

Companion to:
  SFT/data/ift_data_2026_04_30_v9_rms.json     (~12k rows after subsample)
Trainer:
  SFT/autotrain/run_abaligned_sft_qwen25_14b_v9.sh   (Phase B input)
Base model:
  Qwen/Qwen2.5-14B-Instruct
Final HF artefact:
  hf://${HF_USERNAME}/athena-cti-sft-qwen25-14b-abaligned-v9

================================================================================
SECTION 0. v9 strategy (why this corpus is shaped this way)
================================================================================

0.1 Why "v9 RMS slice" exists as a separate manifest
----------------------------------------------------
The v8 14B checkpoint regressed athena-rms (catalog collapse on Phase B),
so v8.1 was authored as a single-pass full-corpus retrain with explicit
RMS catalog floors. v8.1 recovered RMS (+8.9 pp F1 over v8) but
regressed CKT/ATE/RCM/CyberMetric vs v8 because v8.1's
stratified_subsample.py --cap 170 cut every non-RMS family from 9-35K
rows down to 1-4K rows each (0.09-0.16x of v7), starving the broad
knowledge surface that drives those benchmarks.

v9 keeps v8's broad-knowledge baseline (Phase A) and grafts the v8.1
RMS catalog drills onto Phase B as a third dataset alongside the
existing JSON and long-context corpora. To do that without depending
on v8.1's output JSON, the RMS templates are extracted to this
self-contained manifest and the v9 dataset is built independently from
the same Neo4j graph through the same pipeline:

    Sophia-CTI-Templates-v9_rms.txt
      --tmpl_docx2json.py-->         Sophia-CTI-Templates-v9_rms.json
      --make_dataset.sh------>       SFT/data/ift_data_2026_04_30_v9_rms.raw.json
      --stratified_subsample.py-->   SFT/data/ift_data_2026_04_30_v9_rms.json

The cap=170 stratified subsample is mostly inert here because every
template in this file is in PRESERVE_FULL_PREFIXES (AB.RMS.*, JS.RMS.*),
i.e. all rows are retained at 100%. The script is still applied for
parity with the v8.1 build recipe and to keep the post-processing
pipeline byte-for-byte identical between the two corpora.

----------------------------

Final volumes after stratified_subsample (cap=170 inert here, all
RMS preserved at 100%) are typically ~12,158 rows -- 10,433 AB.RMS
catalog/scenario rows + 1,725 JS.RMS JSON-shaped rows.

0.3 Training shape (Phase B graft)
----------------------------------
The v9 launcher consumes this dataset as the third entry in Phase B
alongside the existing v8 JSON and long-context corpora:

  --dataset ift_data_2026_04_29_json_v8,
            ift_data_2026_04_29_longctx_v8,
            ift_data_2026_04_30_v9_rms

Phase B settings (cutoff 16384, packing off, lr 5e-6, group_by_length
on, eff. batch 8) are unchanged from v8 -- the RMS rows are short
relative to the long-context cutoff and ride alongside without
displacing the JSON/longctx exposure. Phase A (cutoff 4096, packing
on, lr 1e-5, the v7 + tulu + alpaca broad-knowledge corpus) is
untouched.

0.4 Build (one-shot, reproducible)
----------------------------------
    # Step 1: compile the manifest into per-template JSON
    python tmpl_gen/scripts/tmpl_docx2json.py \
        -i tmpl_gen/templates/04302026/Sophia-CTI-Templates-v9_rms.txt \
        -o tmpl_gen/data_generation/Sophia-CTI-Templates-v9_rms.json \
        --count_limit 1500

    # Step 2: drive iftgen.py per-template (handled by make_dataset.sh)
    bash tmpl_gen/data_generation/make_dataset.sh \
        tmpl_gen/templates/04302026/Sophia-CTI-Templates-v9_rms.txt \
        _v9_rms_build/triples \
        SFT/data/ift_data_2026_04_30_v9_rms.raw.json \
        10 1500

    # Step 3: stratified subsample (cap=170, RMS preserved at 100%)
    python tmpl_gen/scripts/stratified_subsample.py \
        --in  SFT/data/ift_data_2026_04_30_v9_rms.raw.json \
        --out SFT/data/ift_data_2026_04_30_v9_rms.json \
        --cap 170

    # Step 4 (informational): verbatim-overlap dedup against the eval suite
    python tmpl_gen/scripts/dedup_against_evals.py \
        --input    SFT/data/ift_data_2026_04_30_v9_rms.json \
        --eval-dir SFT/test/benchmark_data \
        --n 13 --hit-threshold 1 \
        --report   SFT/data/ift_data_2026_04_30_v9_rms.dedup.json

The dedup report is informational under the v8 program's structural-
contamination posture (see tmpl_gen/templates/04302026/README.md
Section 2.1 for the full audit trail). The catalog-recall templates
in this file intentionally share short MITRE catalog phrases with
athena-cti-rms.jsonl by design.
