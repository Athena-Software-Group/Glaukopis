================================================================================
Sophia CTI Templates -- v17.1 (Qwen2.5-14B-Instruct, CyberSOCEval-shape narrow
                                 specialist; data-fix recovery of v17, single
                                 variable change vs v17: Shuffle: mcq_multi
                                 added to every template, manifest body
                                 otherwise byte-identical to v17.)
================================================================================

This file is the v17.1 recovery manifest. It addresses a single, scoped
defect in the v17 corpus: every JS.CSE.* template hard-coded the answer
letters into the Answer block (e.g. `"correct_answers": ["A", "B"]`) and
declared no Shuffle directive, so the rendered triples carried only four
unique correct_answers labels across the entire 16,548-row corpus
(["A","B"]=50.7%, ["C"]=24.3%, ["A","B","D"]=18.5%, []=6.6%; letters
E/F/G/H never appeared). This is structural mode-collapse: the model can
satisfy training loss almost entirely by emitting "A,B" regardless of
input, which provides no opportunity to learn CSE multi-select reasoning
from the corpus and is the most plausible single cause of the v17 ->
v16 regression observed across all CTI axes (see _v17_build/v17_post_
mortem analysis and the diagnostic histograms reproduced in
tmpl_gen/templates/05102026/v17_1_plan.txt §2).

v17 -> v17.1 deltas (manifest only; no recipe change)
-----------------------------------------------------
  (1) Every JS.CSE.* template now declares `Shuffle: mcq_multi`.
      The new shuffle mode is implemented in tmpl_parser._shuffle_mcq_
      options_multi (additive change; v8/v12/v15/v16 single-select
      `Shuffle: mcq` path is untouched). On each rendered triple the
      shuffler permutes the variable-size option block (A-H), parses
      the JSON-letter-set answer marker (wrapped <json_object>{...}
      </json_object> or bare {...}), and rewrites the correct_answers
      list to the new positions of the originally correct options.
      Empty correct_answers lists (negative-discrimination templates)
      are preserved.

  (2) NO body changes. Every Instruction, Question, Answer, force,
      Sample, Count, Per_primary_grouping, and Source line is byte-
      identical to v17. The Count: budget is unchanged so the
      v17.1 corpus size lands in the same neighbourhood as v17,
      keeping the SFT step-count comparable for direct attribution.

  (3) NO architecture change. The chained-SFT recipe (v9 narrow drill
      off the v16 base) is preserved verbatim in
      SFT/autotrain/run_sft_qwen25_14b_v16_plus_v17_1_cse.sh. v17.1
      isolates the data-quality variable so the resulting checkpoint
      tells us, unambiguously, whether v17's CTI regression was caused
      by the corpus or by the chained-SFT mechanism.

  (4) NO TAA/CANON/RMS/MCQ carry-over. As in v17 the chained SFT
      preserves the v16 head for those axes via base-model weights;
      the v17.1 shard is single-shape multi-select letter-set MCQs
      only, derived entirely from athena-cti-db-internal.

  (5) Estimated v17.1 declared total: ~22,500 rows, identical to v17.
      Estimated actuals after taa_actor_balance (no-op for JS.CSE.*
      shortnames) and dedup_against_evals: ~16-17K rows. The new
      letter_balance_gate.py adds a build-time check that letters
      A-E each carry >=8% of correct slots, no single combo carries
      >15% of rows, and unique-combo count >=50.

  (6) ZERO CONTAMINATION. As in v17 the detonation-report and intel-
      report PREFIXES are entirely synthetic, generated from MITRE
      ATT&CK relationships in the local athena-cti-db Neo4j. The
      licence-allowlist gate (carried from v13/v14/v16/v17) validates
      every emitted row carries source=athena-cti-db-internal.

================================================================================
Source: athena-cti-db-internal
Anchor selector {mw:malware.name} binds to MITRE ATT&CK malware/software
(~830 documented entries in the local Neo4j; ~9,200 malware->attack-pattern
edges via the `uses` relationship). The synthetic detonation report prefix
is JSON-shaped with vx_family, sha256 (literal "hash" -- the real CSE-MAL
benchmark strips hashes via _strip_hashes before scoring), and a
mitre_attcks list seeded from the malware's documented attack-pattern set.

against the multi-choice options, and emit the bare {"correct_answers":
[...]} JSON shape that the CSE-Malware extractor parses (no <json_object>
wrapper for this family; the bare-brace path is the malware extractor's
primary parse hook -- see SFT/test/pipelines/post_processing/cybersoceval.py).
